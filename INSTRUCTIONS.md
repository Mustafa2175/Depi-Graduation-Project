# 🚀 دليل مشروع NileTech Pulse: المرحلة الثالثة
## نظام تنظيف وتوحيد بيانات سوق العمل التقني المصري

هذا الملف يشرح كيفية تشغيل نظام معالجة البيانات (Processing Layer) الذي يقوم بتحويل البيانات الخام إلى بيانات نظيفة وموحدة جاهزة للتحليل.

---

## 🏗️ هيكل البيانات (المعمارية)
المشروع يتبع نظام **طبقات البيانات (Medallion Architecture)**:
1.  **الطبقة الخام (Bronze):** في `data/raw/` - تحتوي على ملفات JSON الأصلية.
2.  **الطبقة الفضية (Silver):** في `data/silver/` - تحتوي على ملفات CSV النظيفة والموحدة.
3.  **الطبقة الذهبية (Gold):** قاعدة بيانات **PostgreSQL** بتصميم **Star Schema** (المرحلة الرابعة ✅) — تُحمَّل عبر `warehouse/` ثم تُحوَّل إلى نماذج تحليلية بـ **dbt** (المرحلة الخامسة ✅).

---

## 📁 شرح مجلد البيانات (`data/`)
هذا المجلد هو "مخزن" المشروع، وينقسم إلى 5 أقسام رئيسية:

1.  **`data/raw/` (الطبقة الخام):**
    *   يحتوي على البيانات الأصلية كما جاءت من الإنترنت بصيغة **JSON**.
    *   مقسم داخلياً حسب الموقع (Wuzzuf, Bayt...) ثم حسب تاريخ اليوم.
2.  **`data/silver/` (الطبقة الفضية):**
    *   يحتوي على البيانات بعد تنظيفها وتوحيدها بصيغة **CSV**.
    *   هذه البيانات هي التي سيتم رفعها لقاعدة البيانات.
3.  **`data/meta/` (بيانات التتبع):**
    *   يحتوي على ملف `manifest.json` الذي يستخدمه النظام ليعرف الملفات التي تمت معالجتها بالفعل لضمان عدم التكرار.
4.  **`data/failed/` (سجل الأخطاء):**
    *   إذا احتوى أي ملف خام على سجلات تالفة (بيانات مفقودة مثلاً)، يقوم النظام بعزلها هنا لمراجعتها يدوياً.
5.  **`data/reports/` (تقارير الجودة):**
    *   يحتوي على تقارير بصيغة **JSON** توضح نتائج عمليات الفحص (عدد الوظائف، نسبة النجاح، التكرارات المحذوفة).

## 🛰️ المرحلة الثانية: جمع البيانات (Producers)
قبل البدء بالتنظيف، يجب أولاً جمع البيانات. كل مصادر الجمع أصبحت في حزمة موحّدة
`producers/`، وكل مُنتِج (producer) يُخرج بياناته بنفس **العقد الموحّد**
(`producers/contract.py`) مهما كانت طريقة الجمع — مما يجعل طبقة المعالجة مستقلة عن المصدر.

### المُنتِجات الخمسة:
| المصدر | الطريقة | يحتاج متصفح؟ |
|--------|---------|--------------|
| **Wuzzuf** | `requests` + BeautifulSoup | لا (مع إثراء وصف الوظيفة من صفحة التفاصيل) |
| **Forasna** | `requests` + BeautifulSoup | لا |
| **Jobzella** | Selenium (headless) | نعم |
| **Bayt** | undetected-chromedriver (**headful**) | نعم — خلف Cloudflare فيحتاج شاشة عرض |
| **Indeed** | `python-jobspy` | (مكتبة) — قد يحتاج بروكسي على سيرفرات السحابة |

### كيفية التشغيل:
```bash
make scrape                              # تشغيل كل المصادر المتاحة
make scrape SOURCES="wuzzuf forasna"     # تشغيل مصادر محددة فقط
python3 -m producers.runner --list        # عرض المصادر المعروفة
JOB_QUERY="data engineer" MAX_JOBS=200 make scrape   # تحديد نطاق البحث والحجم
```

*   **ملاحظة:** المُنتِجات المعتمدة على المتصفح (Bayt, Jobzella, Indeed) تُتخطّى تلقائياً
    بأمان إذا لم يتوفر متصفح/شاشة عرض، بينما تستمر المصادر المعتمدة على `requests` وكامل
    المسار اللاحق بالعمل (بما في ذلك داخل Docker).
*   البيانات تُحفظ تلقائياً في `data/raw/` مقسمة حسب اسم المصدر وتاريخ اليوم.

## 🛠️ المرحلة الثالثة: كيفية التشغيل
بمجرد جمع البيانات الخام، يمكنك البدء بعمليات التنظيف والفحص:

### 1. المتطلبات الأساسية
تأكد من تجهيز البيئة (إنشاء `.venv` وتثبيت كل الاعتمادات):
```bash
make setup
```

### 2. معالجة البيانات (Cleaning & Standardization)
لتشغيل عملية التنظيف لجميع المصادر دفعةً واحدة (مُنظِّف واحد مستقل عن المصدر):
```bash
make pipeline      # أو: PYTHONPATH=. .venv/bin/python -m processing.main
```
بوابة الجودة مدمجة داخل المعالجة: تُكتب السجلات السليمة إلى `data/silver/`،
وتُعزل التالفة في `data/failed/`، ويُكتب تقرير لكل تشغيل في `data/reports/`.

---

## 🧩 المكونات الأساسية للنظام

### 1. عقد البيانات (`processing/core/schema.py`)
يحدد الحقول الإجبارية (مثل مسمى الوظيفة، الشركة، الرابط) ويضمن توحيدها في كل المصادر.

### 2. الأدوات المساعدة (`processing/utils/`)
- **`cleaners.py`:** مسح نصوص HTML والرموز من العناوين.
- **`parsers.py`:** تحويل التواريخ النسبية (منذ 3 أيام) إلى تواريخ حقيقية واستخراج الرواتب.
- **`mappings.py`:** **المحرك الأساسي** لتوحيد أسماء المدن والمحافظات بالعربي والإنجليزي.
- **`skills.py`:** استخراج المهارات والتقنيات (Python, SQL, Airflow...) من عنوان ووصف الوظيفة عبر مطابقة الكلمات المفتاحية، وإخراجها في حقل `skills` موحّد.
- **`classifiers.py`:** تصنيف نمط العمل `work_mode` (remote / hybrid / on-site) ونوع التوظيف `employment_type` (full-time / part-time / contract / internship / freelance).

### 3. المُنظِّف الموحّد (`processing/cleaner.py`)
مُنظِّف واحد مستقل عن المصدر يحوّل أي سجل بالعقد الموحّد إلى المخطط القياسي
(`JobSchema`) — لا يوجد منطق خاص بكل موقع، لأن المُنتِجات تُخرج جميعها نفس الشكل.

## 🔍 الجودة وإزالة التكرار
- **إزالة التكرار (Deduplication):** النظام يولد "بصمة رقمية" (`job_hash`) لكل وظيفة لمنع تكرارها حتى لو جاءت من مواقع مختلفة.
- **التشغيل الذكي:** النظام يتذكر الملفات التي عالجها عبر ملف `manifest.json` ولا يعيد معالجتها إلا إذا قمت بحذفه.
- **بوابة الجودة (Quality Gate):** مدمجة داخل `processing/main.py` — يتم فحص كل سجل بواسطة `DataValidator`، وتُكتب السجلات السليمة فقط إلى `data/silver/`، بينما تُعزل السجلات التالفة في `data/failed/` مع كتابة تقرير لكل تشغيل في `data/reports/`.

---

## 🏛️ المرحلة الرابعة: النمذجة وقاعدة بيانات PostgreSQL (Star Schema)
يتم الآن تحميل البيانات النظيفة من `data/silver/` إلى قاعدة بيانات **PostgreSQL**
بتصميم **Star Schema** (طبقة Gold). كل الأكواد في مجلد `warehouse/`.

### المكونات:
- **جدول الحقائق:** `fact_job_postings` (صف واحد لكل وظيفة فريدة، إزالة التكرار عبر `job_hash`).
- **جداول الأبعاد:** `dim_date`, `dim_source`, `dim_location`, `dim_company` (بنظام **SCD Type 2** لتتبع تغيّر بيانات الشركات), `dim_job_category`, و `dim_skill` مع جدول الربط `bridge_job_skill`.
- **جداول مرجعية:** `ref_governorate` (محافظات مصر) و `etl_load_log` (تتبع الملفات المحمّلة للتحميل التزايدي).

### كيفية التشغيل:
```bash
export PYTHONPATH=$PYTHONPATH:.
export PGDATABASE=job_market_tracker          # عدّل متغيرات PG* حسب إعداداتك
createdb job_market_tracker                    # أول مرة فقط

# إنشاء المخطط + البيانات المرجعية ثم التحميل
python3 warehouse/load_to_postgres.py --init

# التشغيل التزايدي لاحقاً (الملفات الجديدة فقط)
python3 warehouse/load_to_postgres.py
```

### الاختبار الشامل (End-to-End):
```bash
make test      # مجموعتا اختبارات المستودع والمسار الكامل
make stress    # اختبار ضغط بحجم كبير (تحميل + مستودع + dbt)
```
الاختبارات **معزولة وآمنة تماماً**: تعمل على قاعدة بيانات مؤقتة
(`<PGDATABASE>_test` تُنشأ وتُحذف في كل تشغيل) ومجلد عمل مؤقت، فلا تمسّ أبداً
بياناتك الحقيقية في `data/` ولا المستودع الفعلي (انظر `warehouse/isolation.py`).
التفاصيل الكاملة في `warehouse/README.md`.

## 🧱 المرحلة الخامسة: طبقة التحويلات باستخدام dbt
يقوم **dbt** بتحويل الـ Star Schema إلى نماذج تحليلية مُختبَرة وموثّقة (طبقة Gold النهائية)
التي ستخدمها واجهة الـ API في المرحلة التاسعة. كل الأكواد في `dbt/job_market_tracker/`.

### الطبقات:
- **staging** (Views): قراءة نظيفة من جداول الـ Star Schema.
- **intermediate** (Views): `int_job_enriched` (دمج الحقائق مع كل الأبعاد) و `int_job_skills`.
- **marts** (Tables): نماذج تحليلية جاهزة — الرواتب، الوظائف الأكثر طلباً، المهارات،
  الشركات، التوزيع الجغرافي، نمط العمل، واتجاهات التوظيف.
- **snapshots**: `snap_company` لتتبّع تغيّر بيانات الشركات (SCD2) عبر dbt.
- **الاختبارات:** 49 اختبار جودة (unique / not_null / accepted_values / relationships).

### كيفية التشغيل:
```bash
make dbt          # تشغيل dbt build (بناء + اختبارات + snapshot)
```
التفاصيل الكاملة في `dbt/job_market_tracker/README.md`.

## 📦 التشغيل المحمول (يعمل على أي جهاز)
لا يوجد أي إعداد ثابت (hardcoded) — كل الإعدادات من متغيرات البيئة / ملف `.env`
(انظر `.env.example`). توجد طريقتان:

```bash
# الطريقة (أ) — Docker (موصى بها، بدون أي متطلبات محلية)
cp .env.example .env
docker compose up --build      # يشغّل PostgreSQL + المعالجة + المستودع + dbt كاملة

# الطريقة (ب) — محلياً (تحتاج uv أو Python 3.12)
make setup                     # إنشاء بيئة 3.12 + تثبيت كل الاعتمادات
make db-up                     # تشغيل PostgreSQL عبر Docker (اختياري)
make bootstrap                 # تجهيز القاعدة + التحميل + dbt (قابل للتكرار بأمان)
```
أوامر مفيدة: `make test` (الاختبارات)، `make stress` (اختبار الضغط)، `make` (عرض كل الأوامر).
